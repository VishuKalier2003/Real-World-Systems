API Gateway / Load Balancer
Infrastructure
What
Single entry point routing HTTP requests to backend services with SSL termination.
Why Separate
Decouples client-facing contracts from service internals, enabling independent scaling later.
Problems Solved
Prevents direct client-to-service coupling, handles TLS, provides first-line rate limiting and request routing for future microservices.
Born in Phase I. Evolves from Nginx to Envoy/Istio by Phase IV.
User & Authentication Service
Core
What
Manages users, passwords, sessions/JWT, and basic role distinction (admin vs member).
Why Separate
Identity is a cross-cutting concern; isolating it prevents auth logic from leaking into business code.
Problems Solved
Centralizes security policy, enables SSO migration in Phase III, and ensures consistent session management across all future services.
Born in Phase I. Grows into full IAM with RBAC, SAML, SCIM by Phase IV.
Incident Management Service
Core
What
CRUD lifecycle for incidents: create, acknowledge, resolve, assign, and severity management.
Why Separate
The heart of the domain; isolation protects incident state logic from notification or scheduling complexity.
Problems Solved
Provides single source of truth for incident state, enables audit trails, and allows independent scaling of write-heavy incident ingestion.
Born in Phase I. Shards by tenant ID in Phase V; adds event sourcing in Phase VI.
Notification Service (Email)
Core
What
Accepts notification requests and dispatches emails via SMTP or transactional email APIs.
Why Separate
Notification delivery is unreliable and slow; decoupling prevents email latency from blocking incident creation.
Problems Solved
Isolates vendor-specific delivery logic, enables retry queues, and supports multi-channel expansion without touching incident logic.
Born in Phase I. Adds SMS in Phase II, Voice/Push in Phase IV, and global routing in Phase VI.
Web Dashboard (React/SPA)
Feature
What
Single-page application for viewing incidents, acknowledging, and configuring basic settings.
Why Separate
Frontend release cycles differ from backend; separation enables independent UI iteration and CDN deployment.
Problems Solved
Provides real-time visibility into system health, reduces MTTR through clear UX, and separates presentation from domain logic.
Born in Phase I. Becomes Progressive Web App in Phase III; adds real-time WebSockets in Phase IV.
PostgreSQL (Primary DB)
Infrastructure
What
Relational database storing users, incidents, and notification logs with ACID guarantees.
Why Separate
Data persistence is infrastructure, not application logic; separation allows swapping storage tech without code changes.
Problems Solved
Ensures durable incident records, supports complex relational queries for incidents-users-services, and provides transactional safety.
Born in Phase I. Replicated read replicas in Phase III; sharded/Citus in Phase V; hybrid with CockroachDB in Phase VI.
Basic Health Monitoring
Monitoring
What
Simple uptime checks, log aggregation, and basic alerting if the platform itself fails.
Why Separate
You cannot rely on the system to monitor itself without potential blind spots during outages.
Problems Solved
Detects deployment failures, provides debugging telemetry, and establishes operational culture before complexity grows.
Born in Phase I. Evolves into full Prometheus/Grafana/ELK stack by Phase III.

What

SMS Gateway
Core
What
Extends Notification Service to dispatch SMS via Twilio or similar carriers with delivery tracking.
Why Separate

Redis Cache Layer
Infrastructure
What
In-memory store for session data, on-call lookups, and rate-limit counters.
Why Separate
Caching needs different scaling patterns and eviction policies than persistent database storage.
Problems Solved

Search & Indexing Service
Infrastructure
What
Elasticsearch/OpenSearch index for full-text search across incidents, logs, and post-mortems.
Why Separate

Problems Solved

Born in Phase III. Scales to federated search across logs and metrics in Phase V.
Audit Logging Service
Compliance

Why Separate

Problems Solved

Born in Phase III. Becomes blockchain-anchored tamper-proof logs in Phase V; cross-region compliance in Phase VI.
Message Queue (Kafka/RabbitMQ)
Infrastructure
What
Durable async pipeline connecting Event Ingestion → Incident Service → Notification Service.
Why Separate
Async middleware is infrastructure with its own operational concerns (partitions, retention, consumer lag).
Problems Solved
Absorbs traffic spikes, enables backpressure handling, guarantees at-least-once delivery, and decouples service availability.
Born in Phase III. Kafka cluster with exactly-once semantics in Phase V; global mirrored topics in Phase VI.
Object Storage (S3)
Infrastructure
What
Stores large payloads: alert attachments, incident images, exported reports, and log archives.
Why Separate

Problems Solved
Offloads binary data from expensive DB storage, provides infinite retention for compliance, and enables CDN distribution.
Born in Phase III. Tiered storage and lifecycle policies in Phase V; multi-region replication in Phase VI.
