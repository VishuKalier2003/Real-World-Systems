Interview Problem Statement

"Design the core ride-hailing matching system for a new Uber-like service launching in a single city. The system must handle riders requesting trips and drivers accepting them in near real-time. We expect 10 concurrent users initially (5 riders, 5 drivers), but the architecture must not preclude future scaling. Riders should see available drivers nearby, request a ride, get matched with the optimal driver, track the trip, and complete payment. Drivers should go online, receive match notifications, navigate to pickup, and mark trip states. You have 45 minutes. Design the data model, API contracts, state machine, and deployment architecture. Highlight consistency requirements, failure modes, and why you choose synchronous vs asynchronous communication."

Must Design
  • • Matching algorithm (naive → optimal)
  • • State machine (Rider/Driver/Trip)
  • • Data schema & persistence
  • • API contracts (REST/WebSocket)
Must Discuss
  • • CAP theorem trade-offs
  • • Idempotency & double-submit
  • • Driver location staleness
  • • Split-brain in matching
Must Avoid
  • • Premature microservices
  • • Over-engineered queues
  • • Ignoring network partitions
  • • Missing idempotency keys

🚨 Critical Design Flaws to Spot

  • • Ride Matcher talks directly to DB — creates tight coupling and horizontal scaling issues.
  • • Single PostgreSQL instance — is a single point of failure; no read replicas.
  • • Redis used for driver locking — if Redis fails, matching stops. No fallback.
  • • Synchronous SMS in critical path — NS failure blocks trip creation.

✅ Correct Design Decisions

  • • API Gateway centralizes auth/routing — allows future service mesh insertion.
  • • Redis for session + pub/sub — correct for 10 users, but needs Sentinel.
  • • Trip Manager separate from Matcher — separates lifecycle from allocation.
  • • Notification Service isolated — allows async conversion later.

Phase I Completion Checklist

Design Artifacts

  • Architecture Decision Records (ADRs) for DB choice, sync vs async
  • Entity-Relationship Diagram (ERD) with all tables, indexes, constraints
  • API Specification (OpenAPI 3.0) for all endpoints with error schemas
  • State Machine Diagram (Trip: REQUESTED→MATCHED→PICKED_UP→COMPLETED)
  • Sequence Diagrams for: Request Ride, Match Driver, Complete Trip
  • Data Flow Diagram (DFD) showing PII boundaries

Knowledge Validation

  • Explain why PostgreSQL over MongoDB for financial/transactional data
  • Calculate DB connection pool size: (10 users × 5 req/s × 50ms) / 1000
  • Define retry strategy with exponential backoff for Stripe API calls
  • Document failure modes: DB down, Redis down, Stripe timeout
  • Security review: SQL injection prevention, XSS, CSRF on admin dashboard
  • Cost estimation: AWS t3.micro × 3, RDS db.t3.micro, ElastiCache cache.t3.micro
