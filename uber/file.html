User & Driver Registry
Core Service
Manages onboarding, profiles, documents, and authentication for riders and drivers.
Why Separate?
Identity lifecycle differs from ride logic; enables independent scaling of auth flows.
Problems Solved
Prevents auth bottlenecks from crashing dispatch; isolates PII for compliance; enables SSO and multi-factor auth independently.
➤ Phase I: Monolith module → Phase II: Independent Auth Service
🎯
Ride Dispatch Core
Core Service
Accepts ride requests, finds nearest drivers, and manages the entire trip lifecycle state machine.
Why Separate?
Most critical path; needs ultra-low latency and independent deployment from billing or analytics.
Problems Solved
Decouples request flow from slow payment gateways; allows geo-based sharding; prevents cascading failures from non-critical features.
➤ Phase I: Monolith module → Phase III: Split into Matching + Location + Trip services
💳
Payment Processor
Core Service
Handles card tokenization, charging, refunds, split payments, and driver payouts securely.
Why Separate?
PCI-DSS compliance scope isolation; payment latency is unpredictable and must not block dispatch.
Problems Solved
Isolates sensitive financial data; enables retry logic independent of ride state; supports multiple payment providers.
➤ Phase I: Stripe SDK wrapper → Phase IV: Full microservice with ledger + fraud detection
🔔
Notification Gateway
Feature
Sends push, SMS, and email alerts for ride status, promotions, and driver assignments.
Why Separate?
Notification delivery is async and unreliable; must not block synchronous ride-matching critical path.
Problems Solved
Decouples user communication from core logic; enables multi-channel fallback; supports templating and localization independently.
➤ Phase I: Twilio/Firebase inline calls → Phase IV: Event-driven with preference engine
🗄️
Primary Database
Infrastructure
PostgreSQL storing users, trips, and payments with ACID guarantees for financial consistency.
Why Separate?
Data persistence is a distinct failure domain; needs independent backup, replication, and scaling strategy.
Problems Solved
Guarantees transactional integrity for payments; enables point-in-time recovery; separates read/write load via replicas later.
➤ Phase I: Single Postgres → Phase III: Read replicas → Phase V: Sharded + Cockroach/Spanner
🌐
API Gateway
Infrastructure
Single entry point for mobile apps; handles SSL termination, rate limiting, and basic routing.
Why Separate?
Client-facing contracts must be decoupled from internal service topology to allow backend refactoring.
Problems Solved
Prevents direct client-to-service coupling; centralizes authentication and throttling; enables A/B testing and canary releases.
➤ Phase I: Nginx reverse proxy → Phase V: Kong/AWS API Gateway with plugin ecosystem

Identity & Auth Service
Infrastructure
JWT issuance, OAuth flows, session management, and multi-factor authentication for all clients.
Why Separate?
Authentication is cross-cutting; centralizing prevents duplication and enables unified security policy enforcement.
Problems Solved
Eliminates auth code duplication across services; enables centralized token revocation; supports social login and enterprise SSO.
➤ Phase II: Extracted from monolith → Phase V: Federated identity with regional providers

Rating & Review Engine
Feature
Collects and aggregates star ratings, written feedback, and tags for both riders and drivers.
Why Separate?
Write-heavy workload with loose consistency requirements; should not burden transactional trip database.
Problems Solved
Isolates write spikes from core trip DB; enables async moderation; supports reputation algorithms without coupling to dispatch.
➤ Phase II: Basic aggregate table → Phase IV: Graph-based reputation + sentiment analysis

Redis Cache Layer
Infrastructure
In-memory caching for session tokens, driver availability status, and frequently accessed user profiles.
Why Separate?
Caching infrastructure has distinct operational needs: eviction policies, memory optimization, and high-availability clusters.
Problems Solved
Reduces database load by 80% for hot data; enables sub-millisecond driver status lookups; prevents thundering herd on launches.
➤ Phase II: Single Redis → Phase V: Clustered with client-side sharding + near-cache

Log Aggregation
Monitoring
Centralizes application logs from all services for debugging, pattern detection, and error alerting.
Why Separate?
Log storage and search are resource-intensive; separating prevents impacting production service performance.
Problems Solved

CDN & Static Assets
Infrastructure

Why Separate?
Static content delivery requires geographically distributed edge nodes, distinct from dynamic API infrastructure.
Problems Solved

Async Message Queue
Infrastructure

Why Separate?
Async messaging has different scaling patterns and failure modes than synchronous request-response services.
Problems Solved

➤ Phase II: RabbitMQ → Phase V: Kafka event streaming with topic partitioning

Matching Engine

Why Separate?

Problems Solved

➤ Phase III: Extracted from monolith → Phase V: Cell-based geo-sharded service with ML scoring

Location Tracker
Core Service

Why Separate?
High write throughput (thousands of GPS pings/sec) with loose consistency needs differs from transactional data.
Problems Solved
