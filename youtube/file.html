6 Expert Answers — 20-Year Google System Architect Perspective

👨‍💼 Senior Staff Engineer, Google Infrastructure | Ex-YouTube Platform Team

Q1: Why separate Auth Service from User Service at this early stage?

Principle: Separation of Concerns at the security boundary. Authentication is a cross-cutting concern with distinct failure modes, scaling patterns, and compliance requirements. At Google, we learned that auth token validation (stateless JWT verification) scales differently than user profile reads (stateful, cacheable). By separating them now, you establish a trust boundary: the Auth Service owns identity verification and token lifecycle, while User Service owns profile data and PII handling. This separation allows independent security hardening—auth can be deployed in a DMZ with stricter network policies. Furthermore, when you later add OAuth2, SAML, or 2FA, you modify only the Auth Service without touching profile logic. The 20-word answer: "Auth is a security boundary with different scaling, compliance, and evolution velocity than user data management."

Q2: How do you handle partial upload failures and resume?

Design: Chunked upload with idempotency keys. The client slices the file into 5MB chunks and uploads each with a X-Upload-Id (UUID generated client-side) and X-Chunk-Index. The server maintains a upload_sessions table tracking received chunks. On failure, the client queries GET /uploads/{id}/status to retrieve missing chunk indices, then resumes. Each chunk is written to a temp directory and checksum-validated (SHA-256). Once all chunks arrive, the server assembles them via cat or stream concatenation into the final object storage path. Idempotency: re-uploading the same chunk index with the same upload ID is a no-op (verified by checksum). At Google scale, this pattern evolved into resumable upload protocols (similar to GCS resumable uploads). Set chunk timeout to 10 minutes and session expiry to 24 hours. Cleanup cron removes orphaned sessions.

Q3: Video formats, validation, and malicious upload prevention?

Defense in Depth. Accept only MP4 (H.264/AAC) and MOV containers—widely supported, well-tested parsers. Validation layers: (1) MIME type whitelist on API Gateway, (2) Magic number verification (file headers, not just extensions), (3) ffprobe metadata extraction (fails on malformed files), (4) ClamAV virus scan on staging area, (5) Container sandboxing—ffprobe runs in a seccomp-bpf restricted container with no network access and 30-second timeout. File size limit: 100MB enforced at nginx (client_max_body_size) AND application layer. Rate limit: 3 uploads per hour per user. Why not accept MKV/AVI? Parser attack surface. At YouTube, we learned that demuxer vulnerabilities are a critical attack vector—stick to formats with mature, audited libraries.

Q4: Database schema design for Users, Videos, Views

PostgreSQL Schema (3NF with strategic denormalization):

users: id (UUID PK), email (UNIQUE, indexed), password_hash (VARCHAR(255), bcrypt), username (UNIQUE), created_at (TIMESTAMP), channel_name, profile_image_url.
videos: id (UUID PK), user_id (UUID, FK→users.id, indexed), title (VARCHAR(255)), description (TEXT), storage_path (VARCHAR(500), UNIQUE), file_size_bytes (BIGINT), duration_seconds (INT), status (ENUM: 'uploading','processing','live','removed'), created_at (TIMESTAMP, indexed), updated_at.
views: id (BIGSERIAL PK), video_id (UUID, FK, indexed), user_id (UUID, FK, nullable for anonymous), watched_at (TIMESTAMP), watch_duration_seconds (INT), ip_address (INET).

Indexes: videos(created_at DESC) for "recent uploads" query; videos(user_id, created_at) for channel pages; views(video_id, watched_at) for analytics aggregation. Constraints: CHECK (file_size_bytes > 0), CHECK (duration_seconds >= 0). Partitioning: Not yet—only needed at >1M rows. Use BRIN index on views(watched_at) when scaling.

Q5: Why store video blobs outside the database?

The BLOB Antipattern. Databases excel at structured query, indexing, and ACID transactions on small records. Video files (100MB+) violate every database optimization assumption: they bloat buffer pools, slow replication (binary log explosion), make backups impractical, and prevent horizontal partitioning (you can't shard a 100MB row evenly). Object storage (S3/MinIO) is designed for immutable, large, sequentially-read blobs with erasure coding, CDN integration, and pay-per-GB pricing. The database stores the pointer (storage_path) to the blob, maintaining the relational model while delegating blob semantics to specialized infrastructure. At Google/YouTube, Colossus (our file system) handles exabytes of video; Spanner (our DB) stores only metadata. This separation is non-negotiable at any scale.

Q6: HTTP Range Requests for streaming

HTTP 206 Partial Content. The client requests Range: bytes=0-1048575 for the first 1MB chunk. Server responds with 206 Partial Content, Content-Range: bytes 0-1048575/104857600, and Accept-Ranges: bytes header. The HTML5 video player automatically issues range requests based on buffer state and user seek position. Server implementation: parse Range header, validate bounds against file size, seek to offset in blob storage, stream exactly requested bytes. Critical: Support multipart/byteranges for Safari compatibility. Set Cache-Control: public, max-age=31536000 on immutable video segments (versioned by encoding). At YouTube, we use DASH/HLS (segmented streaming), but for Phase I, raw MP4 with range requests is sufficient and simpler.

Q7: Caching strategy at 10 users—premature?

No. Caching is an operational necessity, not a performance luxury. Even at 10 users, repeated views of the same video (creator checking their upload) generate redundant disk I/O. Implement a two-tier cache: (1) Nginx proxy_cache for video segments (1GB, LRU), (2) PostgreSQL shared_buffers tuned to 2GB (25% of RAM). The cost is near-zero (RAM is already provisioned), but the benefit is protection against accidental thundering herd when the creator shares their video on Twitter and hits spike to 50 concurrent viewers. Google's rule: always design for 10x the expected peak. Caching is not premature optimization; it's capacity planning. However, avoid distributed cache (Redis) at this stage—unnecessary network hop. Local in-memory and nginx disk cache suffice.

Q8: Rate limiting and abuse prevention

Token Bucket per user IP and user ID. Nginx limit_req_zone for IP-based throttling: 10 req/sec burst=20. Application-layer rate limiting via middleware: Redis-counter (or in-memory Map if no Redis) tracking upload count per user. Rules: 3 uploads/hour, 100 views/minute per IP, 1GB total storage per user. Why both layers? Network-layer blocks DDoS before application logic; application-layer enforces business rules. Return 429 Too Many Requests with Retry-After header. At Google, we use global quota service (Sawzall-based) with hierarchical limits: per-user, per-IP, per-region, per-service. For Phase I, in-memory sliding window is sufficient, but design the interface so you can swap in Redis later without changing controllers.

Q9: Docker vs bare metal deployment

Docker Compose. Always. At 10 users, operational complexity is your primary constraint, not raw performance. Docker provides: (1) reproducible builds—works on your laptop exactly as in production, (2) dependency isolation—PostgreSQL 15 won't conflict with system packages, (3) rollback capability—docker-compose up -d --build with image tags, (4) path to scaling—when you reach Phase III, migrating Docker containers to Kubernetes is mechanical, not architectural. Bare metal requires manual package management, version conflicts, and irreversible system mutations. The performance overhead of Docker (<2% CPU, <1% latency) is negligible compared to developer velocity gains. At Google, everything runs in Borg containers; we haven't deployed to bare metal since 2006. Containerization is the default, not an optimization.

Q10: Essential monitoring at 10 users

The Four Golden Signals + Business Metrics. Even at 10 users, you need: (1) Latency: p50/p95/p99 of upload and streaming APIs (Prometheus histograms), (2) Traffic: requests per second per endpoint, (3) Errors: 5xx rate, upload failure rate, auth rejection rate, (4) Saturation: CPU%, memory%, disk I/O, disk free space. Business metrics: video upload success rate, time-to-first-byte (TTFB) for playback, user registration funnel (started→completed). Tooling: Prometheus + Grafana (free, Dockerized). Alerting: PagerDuty/Opsgenie integration for disk>85% and 5xx>1%. Why monitor 10 users? Because you are learning your baseline. Without baseline, you cannot detect anomalies when scaling to 100 users. At YouTube, every microservice emits thousands of metrics from day one; we delete metrics, not add them later.

Q11: Concurrent uploads and idempotency

Idempotency via client-generated UUID. The client generates an X-Idempotency-Key: UUID header before starting upload. Server maintains an idempotency_keys table (key, created_at, response_status, response_body). On duplicate request: return cached response if within 24 hours. For concurrent identical uploads: use PostgreSQL INSERT ... ON CONFLICT DO NOTHING or distributed lock (Redis SET key NX EX 60). The lock ensures only one upload proceeds; the second waits or receives "upload in progress" status. This prevents duplicate storage consumption and metadata corruption. At Google, idempotency is mandatory for all mutating APIs; we use Spanner's transactional guarantees for global uniqueness.

Q12: Upload API contract design

RESTful with multipart support:

POST /api/v1/videos/initiate → Returns {upload_id, max_chunk_size, expires_at}. Headers: Authorization: Bearer {jwt}.
PUT /api/v1/videos/chunk/{upload_id}/{chunk_index} → Body: binary chunk. Headers: Content-Range, X-Checksum-SHA256. Response: 200 OK, {received_chunks, total_chunks}.
POST /api/v1/videos/finalize/{upload_id} → Triggers assembly. Response: 201 Created, {video_id, status, playback_url}.
GET /api/v1/videos/{video_id}/status → Polling for processing status.

Error codes: 400 (malformed), 401 (unauthorized), 413 (too large), 422 (invalid format), 429 (throttled), 500 (server error), 503 (storage full). All responses include X-Request-Id for tracing. Content-Type: application/json except chunk endpoints (application/octet-stream).

Q13: Disk at 95% capacity

Automated triage with human escalation. Alert fires at 85% (warning) and 95% (critical). Automated response: (1) Stop accepting new uploads (return 503 with reason), (2) Trigger garbage collection of incomplete uploads older than 7 days, (3) Compress access logs older than 30 days, (4) Move cold videos (>90 days unviewed) to cold storage tier if configured. Human response: provision additional block storage (EBS volume attach) or enable S3 lifecycle transition. Never auto-delete user content without explicit policy communicated in ToS. At Google, we use Colossus's automatic rebalancing; at your scale, manual volume expansion is acceptable but must be documented in runbook. Design for 10x headroom: if 10 users generate 10GB, provision 100GB minimum.

Q14: Data durability and backup on single server

3-2-1 Rule adapted for cloud. 3 copies, 2 media types, 1 offsite. Implementation: (1) Primary data on EBS volume with snapshot policy (daily, 7-day retention), (2) PostgreSQL streaming replica on separate volume (logical replication), (3) Weekly pg_dump to S3 Glacier Deep Archive (offsite). Object storage: MinIO with erasure coding (if multi-drive) or daily aws s3 sync to remote S3 bucket. RPO (Recovery Point Objective): 24 hours (daily backup). RTO (Recovery Time Objective): 4 hours (restore from snapshot + replay WAL). Test restores quarterly. At Google, we achieve RPO=0 via synchronous replication across three datacenters; at your scale, accept async replication and document the risk.

Q15: SSL/TLS termination point

Terminate at API Gateway (Nginx). Reasons: (1) Centralized certificate management—only one cert to renew, (2) Backend services communicate over Docker internal network (trusted), avoiding TLS overhead, (3) Nginx is optimized for TLS handshake (session resumption, OCSP stapling), (4) Simpler debugging—plaintext logs inside network. Use Let's Encrypt (certbot) with auto-renewal cron. Cipher suite: TLS 1.3 only, ECDHE with X25519, AES-256-GCM. HSTS header: max-age=31536000; includeSubDomains; preload. Internal service-to-service communication: skip TLS for Phase I (Docker network isolation), but design headers for future mTLS (service account tokens). At Google, we terminate at GFE (Google Front End) with global cert distribution; internal traffic uses ALTS (Application Layer Transport Security), not TLS.
