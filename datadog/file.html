Datadog Architecture Evolution

Microservices roadmap from 0 to 1,000,000 users

Phase I: Foundation & Core Telemetry

0 → 10 Users

Build the absolute minimum viable monitoring platform. Single-tenant mindset, monolithic-adjacent services, manual onboarding. Focus: get data in, store it, visualize it.

M
Metrics Ingestion Pipeline
Core

What is it?

Collects time-series metrics from agents, buffers via queue, validates schema, and routes to storage layers efficiently.

Why separate?

Decouples collection from storage, allowing independent scaling of ingestion throughput versus query workload.

Problems solved

Prevents data loss during spikes, normalizes diverse metric formats, enables backpressure handling, and isolates write hotspots from read paths.

T
Time-Series Database
Core

What is it?

Specialized store for timestamped metric data with high compression, fast range scans, and downsampling capabilities.

Why separate?

Generic databases cannot handle write-heavy sequential loads and time-range queries cost-effectively at scale.

Problems solved

Solves high-cardinality storage, reduces disk footprint via compression, enables sub-second aggregation over millions of active series.

Q
Query Engine / API
Core

What is it?

RESTful interface translating user requests into optimized database queries with caching and basic aggregation.

Why separate?

Isolates query complexity from storage nodes, enabling independent optimization of read latency and API versioning.

Problems solved

Abstracts storage schema changes from clients, prevents expensive ad-hoc queries from crashing ingestion paths.

UI
Web Dashboard UI
Feature

What is it?

Single-page application rendering charts, host maps, and metric explorers consuming the Query API directly.

Why separate?

Decouples presentation layer from backend, enabling independent frontend deployments and A/B testing.

Problems solved

Provides intuitive visualization of raw metrics, reduces time-to-insight, and democratizes monitoring for non-engineers.

A
Identity & Auth Service
Infrastructure

What is it?

Handles user registration, JWT issuance, session management, and basic password-based authentication flows.

Why separate?

Centralizes security concerns, ensuring consistent auth enforcement across all services without code duplication.

Problems solved

Eliminates auth sprawl, simplifies audit compliance, and provides a single revocation point for compromised sessions.

D
Host Agent Daemon
Infrastructure

What is it?

Lightweight process deployed on customer hosts collecting system metrics, dogstatsd packets, and forwarding upstream.

Why separate?

Runs outside the cloud boundary on customer infrastructure, requiring autonomous operation and minimal resource use.

Problems solved

Brings visibility into customer environments without firewall changes, buffers during network partitions, and auto-discovers services.

L
Log Ingestion Pipeline
Core

What is it?

Accepts log streams via TCP/HTTP, applies parsing rules, extracts structured fields, and enriches with metadata tags.

Why separate?

Log traffic patterns differ from metrics: bursty, high volume, requiring different buffering and parsing strategies.

Problems solved

Handles multi-line stack traces, normalizes unstructured text, enables full-text search, and correlates logs with metric time windows.

E
Log Storage & Index
Core

What is it?

Inverted index store optimized for text search, field aggregations, and high-write throughput with configurable retention.

Why separate?

Time-series and text storage have fundamentally different access patterns, indexing overhead, and compression algorithms.

Problems solved

Enables grep-speed search over terabytes, field-based faceting, and cost-efficient tiered storage for compliance retention.

⚡
Alerting Engine
Monitoring

What is it?

Continuously evaluates metric and log thresholds, anomaly conditions, and composite multi-signal alert definitions.

Why separate?

Alert evaluation requires streaming window computation and stateful tracking distinct from ad-hoc user queries.

Problems solved

Reduces mean-time-to-detection, prevents alert fatigue via deduplication, and ensures 24/7 automated monitoring coverage.

N
Notification Router
Infrastructure

What is it?

Dispatch service sending alerts via email, Slack, PagerDuty with retry logic, rate limiting, and escalation policies.

Why separate?

Notification delivery is unreliable by nature; isolating retries and third-party failures from core evaluation is critical.

Problems solved

Guarantees at-least-once delivery, buffers during provider outages, and centralizes contact preference management.

#
Metadata & Tagging Service
Infrastructure

What is it?

Central registry mapping hosts, containers, and services to dimensions, tags, and ownership for query enrichment.

Why separate?

Metadata changes frequently and must propagate to all services without coupling every component to the discovery source.

Problems solved

Enables dynamic filtering by environment, maintains consistency across metrics/logs/traces, and powers cost allocation.

📊
Dashboard Builder
Feature

What is it?

Drag-and-drop interface for creating, saving, and sharing custom dashboards with templated variables and widgets.

Why separate?

Dashboard configuration is a write-heavy user activity that should not impact the read-heavy metric serving path.

Problems solved

Standardizes runbook visualization, enables team-specific views, and reduces repetitive chart creation for common infrastructure.

Trace Storage & Index
Core

What is it?

Columnar store for spans and traces supporting traceID lookup, service graph aggregation, and error rate calculations.

Why separate?

Trace data is write-heavy with rare random reads by traceID, needing different partitioning than time-series or logs.

Problems solved

Stores high-cardinality span attributes, powers service topology maps, and retains critical traces for post-mortem debugging.

AI
Anomaly Detection Engine
Analytics

What is it?

Statistical ML service learning baseline patterns to flag deviations without manually configured static thresholds.

Why separate?

Model training and inference are compute-intensive batch jobs that must not interfere with real-time serving paths.

Problems solved

Eliminates threshold maintenance toil, catches unknown-unknown failures, and adapts to seasonal usage patterns automatically.

G

Infrastructure

Why separate?

Problems solved

Prevents noisy-neighbor resource exhaustion, enables graceful degradation, and provides a unified observability hook for all requests.
