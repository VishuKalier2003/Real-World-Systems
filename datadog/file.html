Phase I: Foundation & MVP

User Scale: 0 → 10 customers | Agents: ~1,000 hosts | Ingestion: 10K metrics/sec
Focus: Single-tenant MVP, basic metrics collection, simple dashboards, manual onboarding. Prove product-market fit with minimal engineering (5 engineers).
Metrics Agent
New Core
Lightweight host daemon collecting system metrics via eBPF and procfs every 10 seconds.

What is it?

Lightweight daemon collecting CPU, memory, disk, and network metrics via eBPF and system calls with minimal overhead.

Why separate?

Decouples collection from storage, allowing independent per-host deployment, versioning, and resource isolation.

Problems solved

Eliminates manual metric instrumentation, reduces observability blind spots, enables real-time infrastructure visibility without code changes.

Time-Series DB
New Core
High-throughput storage engine optimized for timestamped metric ingestion and fast aggregation queries.

What is it?

Specialized database storing timestamped metric tuples with tag-based indexing and aggressive compression algorithms.

Why separate?

Isolates storage concerns, enabling specialized indexing, compression, and query optimization distinct from general OLTP.

Problems solved

Solves high-cardinality metric storage at scale, reduces query latency from seconds to milliseconds, prevents storage bottlenecks.

Query Engine
New Core
Distributed query processor translating DSL requests into optimized time-series scans and aggregations.

What is it?

Distributed processor translating user queries into optimized time-series scans, aggregations, and tag-filtered retrievals.

Why separate?

Abstracts query complexity, allowing multiple frontends to share optimized retrieval logic and caching strategies.

Problems solved

Enables sub-second dashboard loads, supports complex rollups, prevents N+1 query storms from overwhelming storage.

Web Portal
New Feature
React-based customer interface rendering dashboards, host maps, and basic metric explorers.

What is it?

React-based customer interface for dashboards, host maps, metric explorers, and basic alert configuration.

Why separate?

Decouples presentation from data, allowing independent UI iteration, A/B testing, and frontend scaling.

Problems solved

Democratizes data access for non-engineers, reduces MTTR via visual correlation, centralizes observability context.

API Gateway
New Infrastructure
Rate-limited ingress routing external requests to internal services with SSL termination.

What is it?

Rate-limited entry point handling SSL termination, request routing, and basic authentication for all external traffic.

Why separate?

Centralizes auth, throttling, and routing, preventing direct service exposure and enabling unified edge policy.

Problems solved

Prevents DDoS and abuse, enables API versioning, provides unified observability for all ingress traffic patterns.

Identity Service
New Infrastructure
OAuth2/JWT authentication with basic tenant isolation and user session management.

What is it?

OAuth2 and JWT-based authentication service providing basic tenant isolation and user session management.

Why separate?

Centralizes security policy, ensuring consistent authentication across all platform entry points and services.

Problems solved

Prevents unauthorized data access, establishes multi-tenancy foundations, simplifies compliance auditing requirements.

Ingestion Router
New Core
Load-balanced data plane accepting agent payloads and routing to storage with buffering.

What is it?

Load-balanced data plane accepting agent payloads, validating schemas, and routing to appropriate storage tiers.

Why separate?

Decouples data arrival from processing, absorbing traffic spikes via buffering and enabling protocol abstraction.

Problems solved

Prevents data loss during traffic bursts, enables protocol abstraction, allows independent scaling of write paths.

Phase II: Telemetry Expansion

User Scale: 10 → 100 | Agents: ~10K hosts | Ingestion: 100K metrics/sec + 50K logs/sec
Focus: Add logs, basic alerting, and dashboard sharing. Introduce message queuing for reliability. Team grows to 15 engineers.
Log Ingestor
New Core
High-volume text collection pipeline parsing unstructured logs into structured events.

What is it?

High-volume pipeline collecting, parsing, and enriching unstructured logs into structured events with extracted fields.

Why separate?

Logs have different volume, format, and retention needs than metrics, requiring distinct processing semantics.

Problems solved

Unifies structured and unstructured telemetry, enables log-to-metric correlation, reduces storage costs via indexing.

Log Indexer
New Core
Inverted-index storage optimized for full-text search and regex querying across billions of events.

What is it?

Search-optimized storage using inverted indices and columnar compression for full-text log querying at scale.

Why separate?

Requires different storage semantics—text search versus time-series aggregation—needing specialized indexing.

Problems solved

Enables grep-at-scale for debugging, supports compliance log retention, accelerates security investigation workflows.

Alert Evaluator
New Monitoring
Stateful rule engine continuously evaluating threshold and anomaly conditions against streaming metrics.

What is it?

Continuous rule engine evaluating threshold, anomaly, and composite conditions against streaming metric windows.

Why separate?

Isolates critical notification path from analytical queries to guarantee delivery and prevent alert delays.

Problems solved

Reduces alert fatigue via stateful deduplication, guarantees SLA for critical notifications, enables complex multi-condition rules.

Dashboard Renderer
New Feature
Server-side tile rendering with client caching for shared dashboard views and templates.

What is it?

Server-side rendering and client-side caching system for visualization tiles, shared views, and dashboard templates.

Why separate?

Separates data fetching from presentation logic, enabling optimized caching and reducing repeated query load.

Problems solved

Reduces repeated query load on storage, enables shareable dashboard links, supports real-time auto-refresh.

Metadata Catalog
New Infrastructure
Centralized entity resolution storing host, container, and cloud resource tags and relationships.

What is it?

Centralized store for host, container, cloud resource tags, attributes, and entity relationship graphs.

Why separate?

Provides unified entity resolution across metrics, logs, and traces, preventing duplicated metadata handling.

Problems solved

Eliminates metric cardinality explosion, enables cross-signal pivoting, maintains accurate entity relationships.

Event Bus
New Infrastructure
Kafka-backed asynchronous messaging backbone for decoupled inter-service communication.

What is it?

Kafka-backed asynchronous messaging backbone enabling event-driven communication between decoupled services.

Why separate?

Prevents synchronous cascading failures, enables event sourcing, and decouples producer from consumer lifecycles.

Problems solved

Absorbs traffic spikes without backpressure, enables replay for debugging, decouples service deployment cycles.

Time-Series DB
Enhanced Core
Partitioning added for multi-tenant isolation, retention policies, and hot/warm storage tiering.

What changed?

Added tenant-aware partitioning, retention policies, and hot/warm storage tiering for cost optimization.

Why now?

100 customers generate enough data volume that single-node storage becomes a bottleneck and cost issue.

Impact

Enables horizontal scaling, prevents noisy neighbor problems, reduces storage costs through automated tiering.

Focus: Distributed tracing, synthetic monitoring, RBAC, and usage-based billing. Team grows to 40 engineers across 4 teams.

New Feature
Auto-instrumentation libraries capturing distributed request traces with minimal code changes.

What is it?

Language-specific auto-instrumentation libraries capturing distributed request traces, spans, and dependencies.

Why separate?

Requires deep language integration and low-overhead sampling independent of backend storage systems.

Problems solved

Pinpoints latency bottlenecks automatically, maps service dependencies, quantifies error impact across distributed systems.

Trace Store
New Core

What is it?

Columnar storage for spans and trace graphs supporting tail-based intelligent sampling and flame graph queries.

Why separate?

Trace data volume and query patterns differ fundamentally from metrics and logs, needing specialized stores.

Problems solved

Enables flame graph visualization, supports intelligent tail-based sampling, retains critical error traces only.

Why separate?

Problems solved

Detects outages before users report them, validates SLAs from geo-distributed locations, benchmarks API latency externally.

Access Control

Granular RBAC with row-level security and team-based data boundaries for enterprise customers.

What is it?

Granular RBAC and row-level security enforcing team-based data boundaries for enterprise multi-team environments.

Why separate?

Centralizes authorization policy as feature surface expands, preventing ad-hoc auth logic in every service.

Problems solved

What is it?

Real-time aggregation pipeline counting billable events: hosts, custom metrics, logs, spans, and API calls.

Why separate?

Financial accuracy requires isolated, auditable counting separate from functional business logic and storage.

Problems solved

Stream Processor
New Infrastructure
Flink-like real-time computation for windowed aggregations, enrichment, and metric derivation.

What is it?

Flink-like real-time computation engine for windowed aggregations, event enrichment, and derived metric computation.

Why separate?

Requires different runtime characteristics than batch or request-response services, needing stream-specific optimization.

Problems solved

Enables real-time anomaly detection, enriches events with reference data, computes sliding window metrics efficiently.

Query Engine
Enhanced Core
Added cross-signal correlation joining metrics, logs, and traces in unified queries.

What changed?

Added cross-signal correlation capabilities joining metrics, logs, and traces within unified query execution plans.

Why now?

Enterprise users need to pivot from metrics to logs to traces seamlessly during incident investigation.

Impact

Reduces MTTR by correlating signals, enables root-cause analysis across telemetry types, improves user workflows.
