🎯 Deep Dive: Phase I + Core Messaging Gateway

System Design Interview Simulation — 0 to 10 Users

📋 0. Interview Problem Statement

Design the backend architecture for a real-time messaging application (WhatsApp MVP) that supports up to 10 concurrent users. The system must support one-to-one text messaging, user registration via phone number with SMS OTP verification, contact discovery from address books, and message persistence with offline delivery. Users should receive messages in near real-time when online, and queued messages must be delivered when they reconnect. Assume this begins as a monolithic deployment, but you must identify service boundaries that will later evolve into independent microservices. Design the data model, API contracts, real-time transport mechanism, authentication strategy, and deployment topology for this phase.

Details to Design:
  • Real-time transport: WebSocket vs SSE vs Long Polling — justify your choice
  • Database schema: users, conversations, messages, and delivery status
  • Message ordering and exactly-once delivery semantics
  • Authentication flow: phone verification, JWT lifecycle, refresh strategy
  • Offline message queueing and replay mechanism
  • CAP theorem trade-offs with your chosen storage technology
  • WebSocket connection state management and horizontal scaling path
  • Monolith-to-microservices migration boundaries and strangler fig pattern

🧠 6. Architect Answers — 20-Year Google SA Perspective
Q1: WebSocket vs SSE vs Long Polling

WebSocket is non-negotiable here. At Google, we learned that SSE is server-push-only; your client needs to send typing indicators, read receipts, and presence heartbeats without opening parallel connections. Long polling at 10 users works but introduces N+1 query problems and 2-5 second latency windows that destroy perceived real-timeness. WebSocket gives full-duplex over a single TCP connection with sub-100ms latency. The threshold to revisit isn't user count—it's connection count. When you hit ~65K concurrent sockets per server (file descriptor limits), you horizontal shard with sticky sessions. At 10 users, a single Gorilla WS handler on a 2-core VM handles this trivially, but you're building the correct foundation.

Q2: Message Ordering Without Global Clock

Never rely on system timestamps. We use conversation-scoped sequence numbers. Each conversation row in PostgreSQL maintains a last_sequence counter. When User A sends to User B, we do: UPDATE conversations SET last_sequence = last_sequence + 1 WHERE id = ? RETURNING last_sequence. This gives us a strict monotonic sequence per conversation. The message is inserted with this sequence. On the client, we maintain a max_received_sequence per conversation; gaps trigger a repair fetch. This is how Google Chat handles ordering—logical clocks scoped to the conversation, not global Snowflake IDs which only guarantee uniqueness, not causality.

Q3: Database Schema Design

Four core tables. users: id (UUID), phone (unique, indexed), name, public_key, created_at. conversations: id, type ('1:1' or 'group'), last_sequence, created_at. participants: conversation_id, user_id, last_read_sequence, joined_at. messages: id (UUID), conversation_id, sender_id, content (encrypted blob), sequence (unique per conversation), client_message_id (UUID, unique per conversation), status ('sent','delivered','read'), created_at. Index on (conversation_id, sequence DESC) for fetch efficiency. The client_message_id with a UNIQUE constraint on (conversation_id, client_message_id) gives you idempotency for free.

Q4: Reconnect After 8 Hours — No Full Scan

Cursor-based pagination using sequence numbers. The client persists its highest known sequence number per conversation locally (SQLite on device). On reconnect, it sends FetchMissedMessages {last_sequence: 42}. The server queries: SELECT * FROM messages WHERE conversation_id = ? AND sequence > 42 ORDER BY sequence ASC LIMIT 100. This is an index-only scan on the composite index. No timestamps, no offsets. If the client has no local state (new device), we use a separate device_sync_checkpoint table that tracks the last global sequence the device acknowledged. This is identical to how Gmail syncs via history IDs.

Q5: Phone Auth & OTP Brute-Force Prevention

Rate limit at multiple layers. Per-phone: max 3 OTP requests per hour, enforced in Redis with TTL 3600. Per-IP: max 10 requests per minute across all phones. OTP is 6-digit, expires in 5 minutes, stored as bcrypt hash (never plaintext). After 3 failed attempts, invalidate the OTP and require re-request. Use Twilio Verify or AWS SNS with fraud detection. JWT access token expires in 15 minutes, refresh token in 7 days (stored hashed in DB). On logout, revoke refresh token and broadcast to all connected devices to drop WS connections. At Google scale, we use FIDO2/WebAuthn, but for MVP, SMS + JWT with Redis-backed revocation lists is correct.

Q6: Duplicate Prevention on Reconnect

Idempotency keys at the client layer. Every message sent from client carries a client-generated UUIDv4 in client_message_id. The database enforces UNIQUE(conversation_id, client_message_id). On insert conflict, the server returns the existing message row (SELECT + RETURNING). The client treats this as success. For non-message actions (delete, edit), use the same pattern. Additionally, the server maintains an idempotency_key table with 24h TTL for non-database actions like push notification triggers. This dual-layer approach ensures exactly-once semantics without distributed transactions at this scale.

Q7: CAP Theorem Trade-Offs

Single-node PostgreSQL chooses CP (Consistency over Availability). In a network partition between app and DB, the app becomes unavailable rather than serving stale data or accepting writes that might be lost. At 10 users, this is the correct choice—message integrity is more important than 99.999% uptime. The failure mode is a 500 error during DB restart; clients retry with exponential backoff. When you scale to Phase IV and shard, you move to Partition Tolerance + eventual consistency for cross-shard operations, but per-shard you remain CP. Google Spanner gives you CAP illusion with TrueTime, but that's Phase VI complexity.

Q8: WebSocket Connection State Storage

In-memory hash map with Redis as backup. The WebSocket Manager maintains map[user_id] = *Conn in process memory for O(1) routing. On server restart, this is lost—acceptable at Phase I because restart takes 5 seconds and clients auto-reconnect with exponential backoff. We also write SET ws:presence:user_id "server_3" EX 60 to Redis so other servers can check if a user is likely online elsewhere. At Phase III, we move to a distributed pub/sub (Redis Streams or NATS) for cross-server broadcast when group chats arrive. Never store socket state in PostgreSQL; the write latency destroys real-time performance.

Q9: Deployment Strategy

Docker Compose on a single EC2 t3.medium with blue-green deployment. Nginx as reverse proxy routes traffic. PostgreSQL runs in a container with a mounted EBS volume for persistence, Redis in another. The app is a Go binary in a Distroless container. CI/CD via GitHub Actions: build image, push to ECR, SSH to server, pull new image, start on port 8081, switch Nginx upstream from 8080 to 8081, drain 8080, stop old container. This gives zero-downtime deploys without Kubernetes complexity. Health checks at /health verify DB and Redis connectivity. At this scale, avoid k8s—the operational overhead is a tax you don't need yet.

Q10: Sequence Diagram — Offline Delivery

Detailed flow: (1) User A sends message with client_msg_id via WSS. (2) Core Messaging inserts to DB with sequence 5. (3) Check Redis presence for User B: returns nil. (4) Insert into pending_messages queue table or Redis list pending:user_B. (5) Trigger Push Relay with message metadata (not content, for privacy). (6) User B's device receives FCM notification, opens app. (7) App establishes WSS, sends FetchMissedMessages {last_sequence: 4}. (8) Server queries messages WHERE sequence > 4, returns message 5. (9) User B app displays message, sends ACK. (10) Server updates status to 'delivered', removes from pending queue. If ACK is lost, client deduplicates via client_msg_id on next reconnect.

Q11: Monolith to Microservices Migration

Strangler Fig Pattern with API Gateway introduction. Step 1: Place Kong/Nginx in front of monolith without changing routes. Step 2: Extract User Service first—it's the most isolated bounded context. Route /auth/* and /users/* to new service, everything else to monolith. Share the PostgreSQL instance initially (no data split yet). Step 3: Extract Media Service when storage costs spike. Step 4: Extract Group Service when fan-out logic becomes complex. Step 5: Finally split Chat Service. At each step, the old monolith code is deleted (strangled). Never do a big-bang rewrite—Google learned this painfully with AdWords. Database-per-service comes last, not first.

Q12: SLIs and SLOs for Phase I

Three SLIs, conservative SLOs. (1) Message Send Latency (p99): SLI = time from WSS send to DB persist. SLO = < 500ms at p99. (2) Delivery Success Rate: SLI = ratio of messages marked 'delivered' to total sent. SLO = > 99.5% within 24 hours (accounts for offline users). (3) API Availability: SLI = ratio of successful HTTP 2xx to total requests. SLO = > 99.9% excluding planned maintenance. No error budget yet at Phase I—just measurement. Use Prometheus + Grafana to track. Alert on SLO burn rate > 10% in 1 hour. These numbers are aspirational but achievable on a single node with proper connection pooling.
